# وثيقة المعمارية وهيكلية المكونات لمتجر مركز الأحمدي للجوالات 📱🚀
**Alahmadi Mobile Center - High-Level Architecture & Components**

توضح هذه الوثيقة المعمارية الهندسية الشاملة لنظام متجر مركز الأحمدي للجوالات ومستلزماتها، والذي يجمع بين ثلاثة أدوار تفاعلية رئيسية (تطبيق العميل، لوحة تحكم الإدارة، وتطبيق مناديب التوصيل) متصلين ببنية سحابية موحدة وقوية تعتمد على **Supabase Cloud Backend**.

## 1. المخطط العام للنظام (High-Level System Architecture)

تم تصميم النظام ليعمل بكفاءة فائقة وتوافق تام مع الهواتف الذكية أولاً (Mobile-First) وشاشات الويب المختلفة، بالاعتماد على الاتصال الفوري بقاعدة البيانات والخدمات السحابية:

## 2. هيكلية المكونات البرمجية وتفاعلها (Component Structure)

يتكون المشروع البرمجي من وحدات مستقلة ومترابطة تضمن سهولة التطوير والصيانة:

### أ. واجهات المستخدم (Presentation & View Layer - Jetpack Compose RTL)
*   **شاشات العميل (Customer Screen Suite):**
    *   `StoreHomeScreen`: يعرض أحدث الهواتف والإكسسوارات مع ميزة فرز العروض والبحث الفوري.
    *   `ProductDetailScreen`: صفحة كاملة لكل مواصفات الجهاز وصوره وتقييمات العملاء.
    *   `CartScreen` & `CheckoutScreen`: معالجة السلة والشراء السريع والدفع عند الاستلام أو التجريبي.
    *   `OrderTrackingScreen`: تتبع حالة الطلب الفوري للزوار والمسجلين بالتوكن الآمن.
*   **لوحة الإدارة (Admin Dashboard Component):**
    *   `AdminDashboardScreen`: تقارير المبيعات، مؤشرات الأداء، مستويات المخزون، والمخططات البيانية.
    *   `ProductManagementScreen`: إضافة، حذف، وتعديل كميات وأسعار السلع.
    *   `OrderManagementScreen`: قبول الطلبات وإسناد الشحنات للمناديب المناسبين.
*   **بوابة المندوب (Delivery Suite):**
    *   `DeliveryHomeScreen`: استعراض الشحنات الجاهزة للتوصيل والمسندة لعهدة المندوب.
    *   `DeliveryMapScreen`: تتبع العنوان السكني للعميل والاتصال به فورياً بلمسة زر واحدة.

### ب. منطق العمل وإدارة الحالة (Business Logic & State - MVVM/MVI)
*   `StoreViewModel`: العقل المدبر للتطبيق، يربط المكونات المرئية بالبيانات، ويدير الحالات الهامة التالية:
    *   `MutableStateFlow<List<Product>>`: حالة الكتالوج الحالي والمحدث فورياً.
    *   `MutableStateFlow<CartState>`: حالة السلة المحسوبة بدقة شاملة الخصومات والضرائب.
    *   `MutableStateFlow<UserRole>`: نظام الصلاحيات النشط (عميل، مدير، مندوب) لتوجيه واجهة المستخدم ديناميكياً.

## 3. طبقة واجهة برمجة التطبيقات المركزية (Centralized API Layer)

لكي تتوحد العمليات البرمجية وتعمل التطبيقات الثلاثة (العميل، الإدارة، المندوب) بانسجام وبأقصى درجات الأمان، يعتمد النظام على **طبقة واجهة برمجة تطبيقات مركزية (Centralized API Layer)** تعمل كجسر وسيط وموجه موثق بين التطبيق وخدمات Supabase السحابية:

```
                  ┌──────────────────────────────────────────────┐
                  │          مكونات الواجهة الأمامية             │
                  │   (تطبيق العميل - لوحة الإدارة - المندوب)     │
                  └──────────────────────┬───────────────────────┘
                                         │
                                         ▼
                  ┌──────────────────────────────────────────────┐
                  │    طبقة واجهة برمجة التطبيقات المركزية (API)  │
                  ├──────────────────────────────────────────────┤
                  │  [AlahmadiApiService] - واجهة الاتصال الموحدة │
                  │  ├─ AuthRepository (إدارة الجلسات والمصادقة)  │
                  │  ├─ ProductRepository (جلب وتعديل السلع)     │
                  │  ├─ OrderRepository (العمليات المالية والطلب)│
                  │  └─ ShipmentRepository (تتبع مناديب التوصيل)  │
                  └──────────────────────┬───────────────────────┘
                                         │  (تنفيذ بروتوكولات REST / WebSockets)
                                         ▼
                  ┌──────────────────────────────────────────────┐
                  │       بنية Supabase السحابية المؤمنة         │
                  └──────────────────────────────────────────────┘
```

### أ. تفاصيل الهيكل والمستودعات المركزية (Repository & API Modules)
1.  **مستودع الهوية والتحقق (`AuthRepository`):**
    *   `signIn(phone, password)`: تسجيل دخول آمن لكل دور (عميل، مدير، مندوب).
    *   `signUp(name, phone, password)`: إنشاء حساب عميل جديد وربطه ذاتياً بجدول الـ Profiles.
    *   `getCurrentUserRole()`: التحقق من الصلاحيات النشطة من خلال رمز الـ JWT الموّقع آمنًا.
2.  **مستودع المنتجات والتصنيفات (`ProductRepository`):**
    *   `getCategories()`: استرجاع جميع التصنيفات لتعبئة شريط التنقل العلوي باللغة العربية.
    *   `getProducts(categoryId, searchQuery, isOffer)`: جلب الكتالوج مصفى ومفرز حسب رغبة المستخدم.
    *   `upsertProduct(product)`: (خاص بالإدارة) إضافة منتج جديد أو تحديث السعر والمخزون.
3.  **مستودع المشتريات والطلبات (`OrderRepository`):**
    *   `createOrder(order, items)`: تسجيل الفاتورة بنظام المعاملة الواحدة (Atomic Transaction) لضمان عدم حدوث تضارب مالي.
    *   `getCustomerOrders()`: استرجاع سجل مشتريات العميل المسجل الحالي.
    *   `trackGuestOrder(token)`: تتبع حالة شحنة العميل الزائر بأمان عبر توكن التتبع الفريد.
4.  **مستودع الشحن والتوصيل (`ShipmentRepository`):**
    *   `getAvailableShipments()`: (خاص بالمناديب) استعراض الطرود الجاهزة للتوصيل.
    *   `updateShipmentStatus(shipmentId, status, notes)`: (خاص بالمناديب) تحديث حالة الشحنة لحظياً.

### ب. بروتوكولات الاتصال والمزامنة (Communication & Serialization Protocol)
*   **اتصال RESTful API:** يعتمد على معيار JSON الموحد للتخاطب مع نهايات Supabase (Endpoints) وبدعم كامل للترويسة الأمنية وحقن توكن المستخدم JWT: `Authorization: Bearer <token>`.
*   **المزامنة الفورية (WebSockets / PostgreSQL Realtime):** تستخدم للاشتراك الفوري بجدول `shipments` و `notifications` لتحديث واجهة العميل عند خروج الشحنة مع المندوب دون الحاجة لإعادة تحميل الصفحة.

## 4. تدفق البيانات والعمليات الأساسية (Core Data Flows)

### أ. دورة حياة الطلب والشحن (Order & Shipment Lifecycle)
1.  **الطلب:** يضيف العميل المنتجات للسلة، ويضغط "تأكيد الشراء". يتم تسجيل الطلب في جدول `orders` وعناصره في `order_items` ويسجل سجل مالي معلق في `payments`.
2.  **التجهيز:** يصل تنبيه فوري للإدارة في لوحة التحكم، تبدأ الإدارة بالتجهيز وتغيير الحالة لـ `قيد التجهيز`.
3.  **التكليف والشحن:** تصدر الإدارة شحنة جديدة في `shipments` وتسندها لمندوب توصيل متاح وتغير الحالة لـ `جاهز للتسليم`.
4.  **التوصيل والرحلة:** تظهر الشحنة في تطبيق المندوب فوراً. يبدأ المندوب الرحلة ويغير حالتها لـ `في الطريق`.
5.  **المزامنة التلقائية:** تقوم وظيفة Postgres التلقائية (`Trigger`) بمزامنة حالة الطلب الأصلي تلقائياً ليكون `في الطريق` ويتم إرسال إشعار فوري للعميل في جدول `notifications`.
6.  **التسليم المالي:** فور تسليم الشحنة للعميل وتأكيد المندوب بالدفع، تتحول حالة الشحنة لـ `تم التوصيل` وحالة الفاتورة لـ `مدفوع` تلقائياً عبر قاعدة البيانات.

```
┌─────────┐      إدراج طلب جديد       ┌──────────┐      إسناد الشحنة       ┌──────────┐
│  العميل  │ ───────────────────────> │  الإدارة  │ ───────────────────────> │  المندوب  │
└────┬────┘                          └──────────┘                          └────┬─────┘
     │                                                                          │
     │ <────────────────────────────────────────────────────────────────────────┘
                       مزامنة تلقائية للحالة وإشعار فوري (RTL)
```

## 4. بنية الأمان ونظام الصلاحيات (Security & RBAC Architecture)

تعتمد معمارية النظام على نموذج أمني متعدد المستويات لضمان خصوصية بيانات العملاء وسرية البيانات المالية لمركز الأحمدي:

### أ. المصادقة والتحقق (Authentication Engine)
*   يتم تسجيل الدخول وتوثيق الهوية عن طريق **Supabase Auth** والذي يصدر رمز أمان مشفر JWT موقّع لحماية الجلسات.
*   بمجرد نجاح التسجيل، يقوم تريجر لقاعدة البيانات بإنشاء ملف تعريفي تلقائي وموثق برقم الجوال في جدول `profiles`.

### ب. سياسات الحماية على مستوى السطر (Row-Level Security - RLS)
تم تفعيل جدار الحماية RLS على جميع جداول النظام لتطبيق سياسات دقيقة:
*   **منتجات وتصنيفات المتجر:** متاحة للقراءة العامة والبحث لجميع الزوار (`SELECT: public: true`)، بينما تعديل البيانات أو إضافة منتجات وصور حكر تام على حسابات الإدارة الموثقة فقط.
*   **الفواتير والطلبات:** لا يمكن لأي عميل استعراض فواتير أو شحنات عميل آخر بفضل فلترة السياسة الديناميكية:
    `customer_id = auth.uid()`
*   **تتبع الزوار بأمان:** لتجنب إجبار العميل على إنشاء حساب (Guest Checkout)، تم توفير رمز تتبع فريد مشفر UUID باسم `tracking_token`؛ تتيح السياسة للزوار قراءة حالة الطلب المقترن بالتوكن الخاص بهم فقط دون قراءة بقية تفاصيل النظام.
*   **بوابة المناديب:** تمنع السياسة الأمنية أي مندوب من الاطلاع على شحنات غير مسندة لعهدته، أو تعديل أسعار وتفاصيل الطلبات العامة.

## 5. ميزات مرنة وحالات واجهة المستخدم (Resilience & UI States)

*   **Offline Resilience (المرونة عند انقطاع الشبكة):** يدعم محاكي الـ API وقاعدة البيانات سجل تدوين دقيق يحفظ طلبات العميل وتجهيز العمليات حتى استعادة الاتصال.
*   **Visual UI States:** تم تزويد جميع مكونات واجهات Jetpack Compose بآلية معالجة ذكية لحالات البيانات:
    *   **Loading State:** تظهر هياكل تحميل حركية متفاعلة (Shimmering Loaders) بألوان الهوية تمنح انطباعاً بالسرعة.
    *   **Empty State:** واجهات معبرة ورسومية لطيفة باللغة العربية عند خلو سلة التسوق أو المفضلة.
    *   **Error State:** رسائل خطأ واضحة ومريحة باللغة العربية مع زر فوري لإعادة المحاولة.
    *   **Success State:** مؤثرات بصرية جذابة تدل على النجاح (مثل واجهة نجاح الدفع وانطلاق الشحنة).
